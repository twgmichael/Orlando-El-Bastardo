# Render PC Windows Boot Kiosk Plan

## Context

The Linux render PCs are dual-purpose machines: during studio work they boot
Ubuntu Server from the external compute drive and run as OEB render/LLM
workers, but they are also Windows gaming PCs. A monitor is always attached.

`render-pc-01` now has a local no-login kiosk display managed by the render
worker automation. It shows a persistent OEB Studio warning that the screen is
safe to turn off but the PC must stay on, worker status, current job, GPU
telemetry, and a slideshow of recent local renders. The live kiosk action is
now a graceful restart button, not an automated boot-to-Windows button. A
human can still use firmware boot order or the firmware boot menu to choose
Windows after restart.

## Discovery

- The kiosk already provides the right human-facing surface because it is
  visible at the attached monitor before desktop login.
- The warning belongs at the top of the page, full width, so it remains the
  first visual instruction: `Safe to turn off screen.` followed by
  `DO NOT TURN OFF PC.`.
- The restart affordance belongs in the top-right area next to the warning,
  using a clear restart symbol and stateful text: `Safe to restart into Game
  Mode` when idle and `Active render in progress` when a render is assigned.
- The worker identity belongs at the bottom of the page, full width, below the
  render preview and system information areas.
- The kiosk runs a local-only status server and browser session on the Linux
  side, so any restart action can be kept off the LAN.
- The dangerous operation is not the button itself; it is granting reboot
  permission. That should be isolated in a narrow privileged helper rather
  than embedded in the browser or general kiosk server.
- The render worker may be running or eligible to claim jobs, so a restart
  action must respect current operational state.
- The current live behavior should be a normal graceful restart. Automated
  one-time boot selection for Windows remains experimental.
- Initial `render-pc-01` inspection showed the current Linux boot is legacy BIOS
  rather than UEFI and `efibootmgr` was not installed. No Windows GRUB entry was
  visible from the initial non-root check. Therefore the feature should be
  deployed with the UI present but the actual boot action disabled until a
  verified Windows boot entry is configured.
- Follow-up `render-pc-01` boot discovery on 2026-07-19 confirmed Ubuntu is
  running from the external SanDisk disk as legacy BIOS GRUB:
  `/dev/sda1` is a `bios_grub` partition and `/dev/sda2` is the ext4 root.
  The internal Windows disk is UEFI/GPT: `/dev/nvme0n1p2` is a FAT32 EFI
  system partition containing `EFI/Microsoft/Boot/bootmgfw.efi`, and
  `/dev/nvme0n1p4` is the NTFS Windows `OS` partition. `efibootmgr -v`
  returns `EFI variables are not supported on this system`, and `os-prober`
  did not report a Windows GRUB entry.

## Research Notes

Two Linux-side approaches are viable, depending on how the PC firmware and
bootloader are configured.

### GRUB next-entry boot

If the Windows boot path appears in GRUB, a helper can use `grub-reboot` with
the Windows menu entry, then reboot. This is familiar and simple, but it
depends on the GRUB menu entry title remaining stable.

Example shape:

```bash
grub-reboot "Windows Boot Manager"
systemctl reboot
```

### UEFI next-boot selection

If Linux and Windows have separate UEFI boot entries, a helper can use
`efibootmgr` to set the firmware's one-time next boot entry, then reboot. This
is the preferred path when available because it avoids matching a GRUB menu
title and is explicitly one-boot-only.

Example shape:

```bash
efibootmgr -n <windows-boot-number>
systemctl reboot
```

The actual Windows boot number must be discovered per PC and stored in host
vars or generated by an Ansible probe.

## Experimental Boot to Windows Path

This path is experimental and hardware-specific. It is not required for the
current kiosk deployment, and it should not be treated as routine automation
until tested carefully on `render-pc-01`.

The underlying issue is that the external Ubuntu environment currently boots
through legacy BIOS GRUB, while the internal Windows installation appears to be
UEFI/GPT. In that state Linux cannot access EFI runtime variables, so
`efibootmgr` cannot discover or select the firmware's one-time Windows Boot
Manager entry.

A possible path is to make the external Ubuntu disk UEFI-bootable:

1. Back up the external Ubuntu disk before partition or bootloader changes.
2. Boot a rescue or installer environment in UEFI mode.
3. Add a small FAT32 EFI System Partition to the external SanDisk disk without
   disturbing the existing Ubuntu root filesystem.
4. Install the UEFI GRUB target for the installed Ubuntu system onto that EFI
   System Partition.
5. Select the external drive's UEFI Ubuntu entry in firmware and confirm Ubuntu
   still boots normally.
6. Verify that `/sys/firmware/efi` exists and `efibootmgr -v` can read firmware
   entries.
7. Identify the Windows Boot Manager `Boot####` entry, then configure
   `oeb_render_worker_windows_boot_entry` with that value.
8. Only after a successful manual validation, enable
   `oeb_render_worker_windows_boot_enabled`.

This route should preserve Linux as the normal boot target and use only a
one-time Windows boot selection. If firmware behavior varies, if the external
disk cannot present a stable UEFI entry, or if Ubuntu fails to boot through the
new path, leave the kiosk boot action disabled.

## Recommendation

Add a small `Restart Worker PC` action to the kiosk, gated by confirmation and
implemented through a narrowly scoped privileged helper. This replaces the
previous `Boot to Game Mode`/Windows boot control.

Recommended flow:

1. Kiosk top-right control shows a restart symbol and `Safe to restart into
   Game Mode` when idle.
2. Clicking opens a full-screen confirmation:
   `Restart render-pc-01? Rendering will stop.`
3. The confirmation calls a local-only endpoint:
   `POST /api/restart`.
4. The kiosk server executes only:
   `sudo /usr/local/sbin/oeb-graceful-restart`.
5. The helper logs the request and runs `systemctl reboot`.

The browser and kiosk server should never run arbitrary boot commands. They
should only be allowed to call the helper.

## Safety Rules

- Bind the restart endpoint to localhost only.
- Require a confirmation screen, not a single-click reboot.
- Disable or block the action while the worker is busy with an active job.
- Show a clear warning if the worker is online and eligible for jobs.
- Use sudoers to permit only the exact helper path for the kiosk service user.
- Log the request and helper execution to journald.
- Keep Linux as the normal default boot target.
- Do not mutate firmware boot order from the kiosk action.

## Decisions

- Replace the Windows boot affordance with a graceful restart affordance.
- Implement the privileged portion as `/usr/local/sbin/oeb-graceful-restart`.
- Keep the kiosk API local-only.
- Remove the legacy Windows boot helper and sudoers drop-in during deployment.
- Prefer UEFI one-time boot via `efibootmgr -n` when the Windows UEFI entry can
  be reliably identified, but keep this in the experimental path rather than
  the live kiosk action.
- Use GRUB `grub-reboot` only as a fallback when the PC's Windows path is
  managed through GRUB.
- Manage the feature through `project-pi-admin` Ansible with host-level
  variables, for example:
  `oeb_render_worker_windows_boot_enabled`.

## Open Questions

- Does each render PC expose Windows as a separate UEFI boot entry, or only as
  a GRUB menu entry?
- What is the discovered Windows boot identifier for `render-pc-01`?
  - Boot file path discovered: `\EFI\Microsoft\Boot\bootmgfw.efi` on
    `/dev/nvme0n1p2`.
  - Firmware boot number not discoverable while Ubuntu is booted in legacy BIOS
    mode; the machine must boot Linux through UEFI before `efibootmgr` can
    reveal the Windows Boot Manager `Boot####` id.
- Should the helper refuse to run when any OEB job is running anywhere, or only
  when this worker has a current job?
- Should the kiosk require a physical keyboard/mouse confirmation, a typed
  phrase, or a timed hold-to-confirm button?

## Proposed Ansible Shape

- Add render worker defaults:
  - `oeb_render_worker_graceful_restart_enabled: false`
  - `oeb_render_worker_graceful_restart_helper_path: /usr/local/sbin/oeb-graceful-restart`
- Add an optional probe/report task for `efibootmgr` output.
- For `render-pc-01`, leave `oeb_render_worker_windows_boot_enabled: false`
  until Linux is booted in UEFI mode or a reliable GRUB Windows entry exists.
- Template `/usr/local/sbin/oeb-graceful-restart`.
- Template a sudoers drop-in allowing the kiosk user to run only that helper.
- Extend the kiosk server with `POST /api/restart`.
- Extend the kiosk page with a restart control, a confirmation UI,
  not-configured disabled state, busy-state guard, and constrained center
  gallery image area that leaves the footer worker name visible.

## Non-Goal

This plan does not automate booting into Windows. That remains hardware and
firmware specific. The immediate live feature is a safe, visible, local way for
a human at a Linux render PC to restart the machine when no render job is
active.
