# GPU forced Blender render plan

## Context

`render-pc-01` is designed to be a GPU render worker. The NVIDIA runtime is
installed, official Blender 5.0.1 is installed, and the JB100 GPU smoke job
proved that Cycles can render on the GTX 1660 SUPER when the script explicitly
selects CUDA.

The failure mode discovered on 2026-07-20 is that a scene render can still run
on CPU even when it is claimed by `render-pc-01`. The worker process, kiosk, and
job state correctly show the render as active, but `nvidia-smi` reports:

```text
GPU utilization: 0%
GPU memory: 32 MiB / 6144 MiB
compute apps: none
```

At the same time, Blender consumes heavy CPU and writes frames. That means the
render is progressing, but not using the GPU. The root cause is that individual
Blender scene scripts can set `CYCLES` without also forcing the Cycles compute
device to CUDA or OptiX. Blender then falls back to CPU.

This is not acceptable for render-PC jobs. If the harness routes a 3D render to
a GPU render worker, GPU rendering should be the default contract, not an
optional per-script habit.

## Goal

All 3D render assets and scenes sent to GPU-capable render PCs should render on
the GPU by default.

The system should:

- force Cycles GPU device selection for render jobs that require
  `gpu.cycles_render`;
- fail fast or clearly downgrade when no GPU device is available;
- record the selected render device in job logs and manifests;
- make CPU fallback explicit, never silent;
- avoid duplicating GPU setup code across every Blender script.

## Non-goals

- Do not require every Mac preview worker to use GPU-specific settings.
- Do not require EEVEE, Workbench, utility scripts, or non-render Blender tasks
  to use CUDA.
- Do not hardcode `render-pc-01` behavior into creative scene scripts.
- Do not treat `nvidia-smi` alone as sufficient proof that Blender is rendering
  on GPU.

## Render Contract

Jobs that require `gpu.cycles_render` must render with Cycles on a GPU device.

The worker should consider a GPU render valid only when Blender reports:

```json
{
  "engine": "CYCLES",
  "cycles_device": "GPU",
  "compute_device_type": "CUDA or OPTIX",
  "devices": [
    {
      "type": "CUDA or OPTIX",
      "use": true
    }
  ]
}
```

For `render-pc-01`, the expected device is:

```text
NVIDIA GeForce GTX 1660 SUPER
```

## Implementation Plan

### 1. Add a shared Blender render-device helper

Create a shared helper under `tools/` or a small importable support package used
by Blender scripts, for example:

```text
tools/oeb_blender/render_device.py
```

The helper should expose:

```python
configure_render_device(
    require_gpu: bool = False,
    preferred_backends: tuple[str, ...] = ("OPTIX", "CUDA"),
) -> dict
```

The helper should:

1. Set `scene.render.engine = "CYCLES"`.
2. Try preferred GPU backends.
3. Call `prefs.get_devices()`.
4. Enable only GPU devices when available.
5. Set `scene.cycles.device = "GPU"`.
6. Return a structured render-device summary.
7. Raise an error if `require_gpu=True` and no CUDA/OptiX device is available.
8. Only fall back to CPU when GPU is not required.

### 2. Make Blender scripts call the helper

All scene and asset render scripts should call the helper before the first
render operation.

Required script families:

- scene render scripts generated by the conversational build loop;
- checked-in scene scripts such as JB100 action scenes;
- asset review render scripts;
- future animation/movie render scripts.

Example usage:

```python
from tools.oeb_blender.render_device import configure_render_device

render_device = configure_render_device(require_gpu=True)
print("OEB_RENDER_DEVICE", render_device)
```

If import path handling inside Blender makes `tools.oeb_blender` awkward, the
worker adapter should inject the repo root into `PYTHONPATH` before launching
Blender.

### 3. Pass GPU intent from the worker adapter

The Blender worker adapter should inspect job capabilities.

If a job requires `gpu.cycles_render`, the adapter should set environment
variables before launching Blender:

```text
OEB_FORCE_CYCLES_GPU=1
OEB_CYCLES_BACKENDS=OPTIX,CUDA
```

Scripts can then call:

```python
configure_render_device_from_env()
```

This keeps creative scripts portable while making the worker contract clear.

### 4. Add a pre-render smoke probe for GPU workers

`oeb_render_worker` deployment should include a Blender CUDA/OptiX probe, not
only `nvidia-smi`.

The probe should run official Blender through the same headless wrapper used by
the worker:

```text
/usr/local/bin/oeb-blender-headless --background --factory-startup --python <probe>
```

The probe should verify that at least one CUDA or OptiX device is visible and
usable. A host should not advertise `gpu.cycles_render` unless this probe
passes.

### 5. Record render-device metadata

Every GPU-targeted render job should leave a traceable device summary.

Minimum outputs:

- log line: `OEB_RENDER_DEVICE {...}`;
- job `output_summary.render_device`;
- optional manifest sidecar for scene renders;
- kiosk-visible signal when available.

This lets the review page and operator tools distinguish:

- GPU render succeeded;
- CPU fallback occurred intentionally;
- GPU render was requested but unavailable.

### 6. Tighten job routing

Final-quality asset and scene renders intended for render PCs should require:

```json
[
  "blender.command_line",
  "gpu.cycles_render"
]
```

or, for review jobs:

```json
[
  "blender.final_render",
  "gpu.cycles_render"
]
```

Draft or Mac-preview jobs may continue using `blender.preview_render` without a
GPU requirement.

## Immediate Fix For JB100 Pirate Escape

Patch `tools/JB100-pirate-escape.py` to call the shared render-device helper
before rendering frames.

Then:

1. Redeploy or rsync source to `render-pc-01`.
2. Submit the scene job with `gpu.cycles_render` in `required_capabilities`.
3. Confirm live GPU usage:

   ```text
   nvidia-smi --query-compute-apps=pid,process_name,used_gpu_memory --format=csv
   ```

4. Confirm the render-device summary reports:

   ```text
   cycles_device=GPU
   compute_device_type=CUDA or OPTIX
   NVIDIA GeForce GTX 1660 SUPER use=true
   ```

## Verification Checklist

- `nvidia-smi` shows Blender as a compute app during render.
- GPU memory rises above the idle display baseline.
- GPU utilization is nonzero while frames render.
- Blender logs include `OEB_RENDER_DEVICE`.
- Job output summary includes render-device metadata.
- Render artifacts continue uploading through the harness.
- Kiosk GPU row matches the active compute state.

## Open Questions

- Should CPU fallback fail the job when `gpu.cycles_render` is required, or
  should it complete with a warning? Preferred answer: fail fast.
- Should `OPTIX` be preferred over `CUDA` on NVIDIA GPUs? Preferred answer:
  prefer `OPTIX`, fall back to `CUDA`, but record the chosen backend.
- Should render quality presets imply GPU requirements? Preferred answer:
  `final` on render-PC jobs should imply GPU; `draft` and `preview` should
  remain policy-driven.
